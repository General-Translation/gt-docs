## 概要

**動的コンテンツ**は一般的に、ユーザー、コンテキスト、環境などに基づいて変化する可能性のあるコンテンツです。
これは、ユーザー、コンテキスト、環境などに関係なく同じままである**静的コンテンツ**とは対照的に存在します。

<Accordions>
    <Accordion title="動的コンテンツと静的コンテンツの違いは何ですか？">
        TL;DR
        * 静的コンテンツは決して変化しません（生の文字列、テキストなど）。
        * 動的コンテンツは変化する可能性があります（名前、メール、時間、言語など）。

        **静的コンテンツとは何ですか？**

        静的コンテンツとは、一般的にユーザーに提供されるバンドルに存在する生のテキストを指します。
        簡単な判断基準として、開発者がソースコードで読むことができるテキストや文字列は静的テキストです。

        例えば、このファイルを考えてみましょう：

        ```jsx title="Landing.jsx" copy

        export default function Landing() {
            return (
                <> Welcome to my app!</>
            );
        }
        ```
        
        「Welcome to my app!」というテキストは、決して変化しないため静的コンテンツです。

        しかし、ユーザーがログインしている場合に応答するようにページを変更したい場合はどうでしょうか：

        ```jsx title="Landing.jsx" copy

        export default function Landing(user) {

            if (user) {
                return (
                    <h1> Welcome to my app, {user.name}! </h1>
                );
            }
            
            return (
                <h1> Welcome to my app!</h1>
            );
        }
        ```
        これらの2つのフレーズは条件付きでレンダリングされていますが、どちらも静的テキストと見なされます。
        判断基準を思い出してください：`landing.jsx`のソースコードを読むことでこのコンテンツを確認できます。

        しかし、`{user.name}`は変化する可能性があるため、動的コンテンツと見なされます。
        `landing.jsx`ファイルを読むだけでは、ユーザーの画面に何がレンダリングされるかを知ることはできません。
    </Accordion>
</Accordions>


## "翻訳するかしないか"
時には、動的なコンテンツを翻訳したい場合もありますが、他の場合はそのままにしておきたい場合もあります。


良い例としては、ユーザーのメールアドレスや名前が挙げられます。
他の例としては、銀行口座の残高やユーザーのSSN（社会保障番号）などがあります。
このような項目は、(1) アプリが異なる言語でレンダリングされる際に翻訳が必要になる可能性が低く、(2) 変動する可能性があります（この場合、ユーザーごとに異なります）。


### 例

```jsx title="Greeting.jsx" copy
import { T, Var } from 'gt-next'

export default function Greeting(name) {
    return (
        <T id='greeting'>
            Hello, <Var>{name}</Var>!
        </T>
    );
}
```

翻訳に関しては、これには2つの利点があります：
1. あらゆる可能な名前に対して翻訳を作成する必要がありません。
    * `<Var>`を使用することで、基本的に次のような1つの翻訳だけを生成します：
        * \`¡Hola, $\{name\}!\`
    * `<Var>`を使用しない場合、固有の名前ごとにオンデマンド翻訳を実行する必要があります：
        * "¡Hola, Alice!", "¡Hola, Bob!", "¡Hola, Charlie!", "¡Hola, David!", ...
2. また、名前自体が翻訳された形に変わることを心配する必要もありません：（例：「¡Hola, Alicia!」、「¡Hola, Roberto!」、...）。

ご覧のように、`<Var>`コンポーネントは、ロケールに関係なく同じままであるべきコンテンツをラップするために使用する必要があります。
このようにして、動的コンテンツのあらゆる可能な値に対して翻訳を作成する必要性を回避できます。


個人情報を`<Var>`コンポーネントでラップすることで、その情報がGeneral Translations APIに送信されないようにすることができます。
<Callout>
    **例外**

    上記の説明に対する例外は、(1) `<Var>`コンポーネント内でネストされた`<T>`コンポーネントを使用する場合（つまり、ネストされた`<T>`コンポーネントの子要素は翻訳されます）、
    または (2) `<Var>`コンポーネントの子要素内で他の手段を通じて意図的にデータがAPIに渡される場合（例：フェッチ呼び出し）です。
    ただし、これは`<Var>`コンポーネントやGeneral Translations APIの意図された使用方法ではなく、そのような使用は読み込み時間とパフォーマンスに悪影響を与える可能性があります。

</Callout>
